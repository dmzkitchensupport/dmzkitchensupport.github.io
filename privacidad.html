01

Responsable del tratamiento

El responsable del tratamiento de datos personales es De La Mora Zumarán Kitchen Support (en adelante "DMZ" o "el Prestador"), con domicilio en Puebla, México, y contacto en mario@delamorazumaran.com.

La plataforma DMZ Audit es operada exclusivamente por DMZ Kitchen Support como herramienta de auditoría operativa para la industria de alimentos y bebidas.

02

Datos que recopilamos

Datos de identidad del operador

  • Nombre completo del auditor o evaluador
  • Correo electrónico (identificador único de acceso)
  • Credenciales de acceso encriptadas (contraseña con hash bcrypt-12)
  • Rol asignado en la plataforma (admin / auditor)

Datos operativos generados en la plataforma

  • Evaluaciones de alimentos: platillos, categorías, calificaciones por criterio
  • Evaluaciones de servicio de salón: criterios F&B, tiempos de servicio
  • Fotografías de evidencia con sello GPS y metadatos de captura
  • Coordenadas GPS en el momento de la captura y envío
  • Tiempos de servicio (hora inicio, hora fin, duración por platillo)
  • Dictámenes, observaciones y firmas digitales del auditor

Datos técnicos de sesión

  • Dirección IP del dispositivo
  • User-agent (tipo de navegador/dispositivo)
  • Timestamp de inicio y cierre de sesión
  • Eventos de navegación dentro de la app (anonimizados)

No recopilamos: números de tarjeta bancaria, documentos de identidad oficiales, datos biométricos, ni información de menores de edad.

04

Infraestructura y almacenamiento

FRONTEND

GitHub Pages — dmzkitchensupport.github.io

Sitio web estático hospedado en servidores de GitHub Inc. (Estados Unidos). No se almacenan datos sensibles en el frontend.

BASE DE DATOS

Neon PostgreSQL — región us-east-1

Base de datos PostgreSQL con cifrado en tránsito (TLS 1.3) y en reposo (AES-256). Row Level Security (RLS) activo. Particionamiento multi-tenant que aísla completamente los datos de cada cliente.

API

Netlify Functions — dmz-audit.netlify.app

Funciones serverless para procesamiento de autenticación y datos. HTTPS forzado con HSTS. Rate limiting via Upstash Redis. Credenciales de acceso nunca expuestas en el frontend.

EMAIL

Resend — notificaciones transaccionales

Envío de reportes de evaluación al administrador. No se utiliza para marketing ni comunicaciones no solicitadas.

Transferencia internacional

Al utilizar la plataforma, el Cliente acepta que sus datos operativos sean almacenados en servidores ubicados en Estados Unidos, bajo los términos de privacidad de GitHub, Neon y Netlify, los cuales cumplen con estándares equivalentes a la normativa mexicana aplicable.

05

Confidencialidad operativa

DMZ Kitchen Support reconoce que toda la información operativa del Cliente — incluyendo nombres de establecimientos, procesos, estándares de servicio, calificaciones y datos del personal evaluado — constituye información confidencial y propiedad exclusiva del Cliente.

Compromisos del Prestador

  • No divulgar, reproducir ni compartir información operativa del Cliente con terceros bajo ninguna circunstancia
  • No utilizar datos del Cliente para desarrollar productos, servicios o análisis para terceros
  • El personal con acceso a la plataforma está sujeto a acuerdos de confidencialidad internos
  • Los datos de cada cliente están aislados a nivel de base de datos y no se mezclan con datos de otros clientes
  • Las evaluaciones y fotografías son accesibles exclusivamente por el administrador autorizado del cliente

Propiedad de los datos

El Cliente es el propietario absoluto de todos los datos operativos generados en la plataforma. DMZ actúa como procesador de datos, no como titular. En caso de terminación del servicio, el Cliente puede solicitar la exportación completa de sus datos en formato JSON/CSV dentro de los 30 días siguientes.

06

Seguridad técnica

ZERO-TRUST PROTOCOL

Autenticación y control de acceso

JWT HS256 con expiración de 15 minutos. Refresh Token Rotation — la reutilización de un token revoca automáticamente todas las sesiones activas. Contraseñas con hash bcrypt cost-12. Rate limiting: máximo 5 intentos fallidos por 15 minutos por IP.

AUDIT TRAIL INMUTABLE

Trazabilidad permanente

Cada acción genera una huella digital inalterable: quién, qué, cuándo y desde dónde. Los registros de audit_logs están protegidos por triggers de base de datos que bloquean UPDATE y DELETE. Ningún usuario, incluyendo administradores, puede modificar o eliminar el historial de auditoría.

MULTI-TENANT

Aislamiento de datos

Arquitectura multi-tenant con Row Level Security (RLS) en PostgreSQL. Cada tenant opera en su propio espacio lógico aislado. Es técnicamente imposible que un cliente acceda a datos de otro cliente.

Cifrado y transporte

  • TLS 1.3 en todas las conexiones cliente-servidor
  • HSTS activo — el navegador nunca permite conexiones sin HTTPS
  • AES-256 para datos en reposo en la base de datos
  • Headers de seguridad: X-Frame-Options: DENY, X-Content-Type-Options: nosniff
07

Evidencia fotográfica y GPS

Las fotografías capturadas en la plataforma son evidencia incorruptible. Cada foto incluye: coordenadas GPS verificadas, timestamp del servidor, identificador del operador y hash de integridad SHA-256.

Cómo se captura la evidencia

  • Las fotos solo pueden capturarse directamente desde la cámara del dispositivo (capture="environment"). No se pueden adjuntar imágenes de galería.
  • Las coordenadas GPS se registran en el momento exacto de la captura
  • La foto, el GPS y el timestamp del servidor se vinculan mediante hash criptográfico
  • No es posible alterar retroactivamente ninguno de estos datos

Almacenamiento de fotografías

Las imágenes se almacenan en formato base64 en la base de datos PostgreSQL, vinculadas al registro de auditoría correspondiente. No se comparten con terceros ni se utilizan para ningún fin distinto al control de calidad operativo.

Las fotografías contienen metadatos de geolocalización. El auditor consiente expresamente la captura de su ubicación al momento de tomar evidencia fotográfica dentro de la plataforma.

10

Derechos ARCO

De conformidad con la Ley Federal de Protección de Datos Personales en Posesión de los Particulares (LFPDPPP), el titular de los datos tiene los siguientes derechos:

  • Acceso: Conocer qué datos personales suyos están en posesión del Prestador
  • Rectificación: Corregir datos inexactos o incompletos
  • Cancelación: Solicitar la eliminación de sus datos del sistema
  • Oposición: Oponerse al tratamiento de sus datos para finalidades específicas

Para ejercer cualquiera de estos derechos, envíe su solicitud a mario@delamorazumaran.com con el asunto "Derechos ARCO" y su identificación. El tiempo de respuesta es de máximo 20 días hábiles.

11

Retención y eliminación de datos

  • Los datos operativos se conservan durante la vigencia del servicio
  • Al terminar el servicio, los datos se conservan 60 días adicionales para exportación
  • Tras 60 días post-cancelación, los datos son eliminados de forma segura e irrecuperable
  • Los registros de audit_logs se conservan por obligación de trazabilidad durante 5 años
  • Los logs de sesión técnicos se eliminan automáticamente después de 90 días
